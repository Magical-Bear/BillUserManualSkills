## 1. 工具门禁

- 正式需求讨论：必须使用 `superpowers`。
- 技术选型和架构分析：必须使用 `Context7` 查询当前版本文档和官方推荐方案。
- UI、后台、报表、大屏、ECharts、图表、布局、视觉设计：必须使用 `ui-ux-pro-max-skill`。
- 缺失工具时先停止对应阶段，列出缺失项，引导用户安装并验证后继续。

## 2. 需求批准门禁

开发前文档至少包含：背景、目标、非目标、用户和场景、最终展示或交付效果、功能要求、异常处理、生产端、桥梁层、存储层、消费端、API、UI、数据库表字段索引、幂等和兼容策略、技术栈、选择依据、安全权限、敏感配置、日志指标链路追踪、测试计划、部署、健康检查、回滚、验收标准、时间窗口、超时、重试、阈值和待确认项。

只有用户明确表达“需求确认”“方案通过”“按方案开发”或同等含义，才能进入开发。

## 3. 自顶向下场景拆解

用户只说场景时，不直接写代码。先从最终展示效果倒推：

1. 展示效果：谁看、看什么、时区、自然日、指标、筛选。
2. UI：布局、ECharts/表格、交互、空状态、错误状态。
3. 消费端：聚合语义、时间粒度、缺失、重复、延迟数据。
4. API：参数、响应、鉴权、错误码、分页、范围和数据量限制。
5. 数据库：原始数据、业务数据、聚合数据、索引、三范式和扩展。
6. 桥梁层：MQTT、HTTP、网关、任务和入库流程。
7. 生产端：字段、单位、采样频率、上报频率、协议版本、离线缓存。

最终输出正向链路：

```text
生产端 → 桥梁/采集层 → 原始数据 → 标准化/聚合 → API → UI
```

## 4. 领域 Skill 路由

- `doctor-bill-software`：产品、软件架构、API、数据库、UI、ECharts。
- `doctor-bill-hardware`：设备、固件、传感器、串口、MQTT Topic。
- `doctor-bill-ai`：Dify、LangChain、RAG、Agent、微调、推理、评估。
- `doctor-bill-ops`：systemd、linger、GitHub Actions、部署、健康检查、回滚。

## 5. 分支、角色和合并门禁

正式开发必须在功能分支。测试通过不等于允许合并。必须用户明确说“验收通过，合并 main”或同等意思后才能合并、push 或发布。

## 6. 最终报告字段

每轮报告：需求范围、依据、使用 Skill、文档版本、修改文件、数据库变化、测试状态、未测范围、风险、变量/阈值位置、Git 状态、部署和回滚状态。
